Overall System Workflow

The Smart Prescription Analysis System is a Streamlit-based application designed to extract medicine information from prescription images and classify each detected medicine according to its medical department. The system supports both single-image and bulk-image processing workflows, allowing users to analyze individual prescriptions or process multiple prescription images in one session. At the beginning of operation, the user launches the application and selects the preferred workflow mode. The user may choose either Single Image mode for one prescription at a time or Bulk Images mode for multiple prescription files. In addition, the user selects a processing mode, such as High Accuracy or Fast, depending on whether the priority is stronger correction and review or shorter response time.

In Single Image mode, the user uploads one prescription image in JPG, JPEG, or PNG format. In Bulk Images mode, the user uploads multiple images of the same supported formats. After uploading the image or images, the user selects the extraction strategy. The available extraction strategies are Use full image, Use auto crop, and Use both and keep better result. When the full-image option is selected, the system processes the original prescription image directly. When the auto-crop option is selected, the system first detects and crops the lower medicine-dense region of the prescription before performing extraction. When the combined option is selected, the system processes both the original image and the cropped image, then chooses the stronger result based on the extracted medicine information.

After preprocessing, the prescription image is sent to a Gemini or Gemma model for text extraction. The model returns structured or semi-structured prescription text. The system then parses the returned text to identify medicine-related lines and isolate prescribed medication names. During this stage, the application removes unnecessary wrappers, formatting artifacts, numbering, and prompt-related text such as “Input text:” or “Input:”. The extracted medicine names are then normalized and cleaned before classification.

To improve recognition quality, the system performs a medicine-name correction stage. It first checks an alias dataset to determine whether a known OCR-distorted form has already been corrected before. If no alias mapping is found, the system applies fuzzy matching against the local medicine dataset to identify a closely related known medicine name. In High Accuracy mode, the application may additionally use Gemini-based correction to repair OCR-distorted or partially unclear medicine names. This step is intended to improve the recognition of local brand names and weak handwritten extractions while preserving valid medicine text.

Once the medicine names are resolved, the system classifies each medicine into one of the supported departments: Cardiology, Ophthalmology, or Unknown. The department classification process is hierarchical. First, the application attempts exact or near-exact matching using the local seed dataset. If a compatible local classifier is available, the system may use the trained model for prediction. If the local logic cannot confidently determine the correct label, the application may use Gemini fallback in High Accuracy mode. If the system still cannot confidently identify a department, the medicine is labeled as Unknown rather than forcing an incorrect department assignment.

After classification, the application creates clean final result rows containing only two essential fields: medicine_name and department. The system then presents the extracted output to the user in tabular form. For additional transparency, the application also provides a review corrections section where the user can inspect the original medicine text, corrected medicine name, name source, department source, and confidence level. An unknown-only review subsection is also available to help the user focus on unresolved or low-confidence items. The system additionally generates a department summary showing the number of detected medicines and the count per department. To improve usability, the application can also show duplicate-image warnings and retain a recent processing history.

The system includes a manual correction mechanism. The user may edit corrected medicine names and department labels inside the review interface. When the user saves these review corrections, the system updates the alias dataset with OCR-to-corrected-name mappings and may also update the local medicine department dataset with newly validated medicine-department pairs. As a result, future extractions become more accurate because the application gradually learns from user-confirmed corrections.

For output generation, the application supports multiple export options. The extracted results can be downloaded as Excel or CSV files and also as PDF files. In bulk processing mode, the application can additionally create a ZIP package containing combined bulk output, summary files, and per-image CSV files. The system also supports a CSV collection mechanism in which the user can append extracted rows into a selected CSV file. If the auto-append option is enabled, future processed images are automatically added to the same CSV collection without requiring manual re-selection each time.

Overall, the system follows a complete workflow that starts with prescription image upload, continues through preprocessing, text extraction, medicine parsing, correction, and department classification, and ends with result review, manual refinement, and structured export. This workflow ensures that the application can be used not only for one-time extraction but also for iterative dataset building, correction tracking, and prescription-based medicine classification in a reusable and scalable manner.
