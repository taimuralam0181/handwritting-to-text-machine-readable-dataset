Complete System Architecture

The Smart Prescription Analysis System is designed as a Streamlit-based intelligent document-processing application that extracts medicine-related information from prescription images and classifies each detected medicine according to a predefined medical department. The system architecture follows a modular design in which the user interface layer, image-processing layer, text-extraction layer, medicine-resolution layer, classification layer, review layer, and export layer work together to produce structured outputs from unstructured prescription images.

At the top level, the architecture begins with the presentation layer, which is implemented using Streamlit. This layer is responsible for all user interaction and workflow control. Through this interface, the user selects the workflow mode, chooses the processing mode, uploads prescription images, selects extraction strategies, reviews extracted outputs, edits corrections, and downloads final results. The presentation layer also provides auxiliary controls such as dataset upload, CSV collection management, auto-append configuration, duplicate-image warnings, progress indicators, and recent-processing history. This layer acts as the central interaction point between the end user and the underlying processing modules.

The second major architectural component is the input and preprocessing layer. This layer receives uploaded prescription images in JPG, JPEG, or PNG format. It reads the raw image bytes, computes file hashes for duplicate detection, and prepares the image for downstream extraction. In addition to using the original uploaded image, the system provides a heuristic auto-cropping module that attempts to isolate the lower, medicine-dense portion of the prescription. This crop-based preprocessing improves the chance of detecting medicine lines when the full prescription contains large amounts of header or metadata text. When the combined extraction mode is selected, both the full image and the cropped image are processed, and the stronger result is retained.

The next architectural layer is the AI-based text extraction module. This module is responsible for converting the prescription image into machine-readable text. It uses Gemini or Gemma model calls through the configured API connection. The model receives carefully designed prompts that instruct it to return structured prescription information or medicine-only content in a controlled format. The system supports fallback prompt strategies so that if the full structured extraction is weak, a medicine-focused extraction prompt may be used to recover the medicine list more reliably. This layer forms the bridge between raw visual prescription input and the internal text-processing pipeline.

Once text has been extracted, the system enters the parsing and medicine identification layer. This module scans the extracted text to detect medicine-related lines, preferably from the prescribed-medications section, but it also supports fallback scanning across the entire extracted text if the expected structure is missing. The parser identifies dosage-like patterns, bullet-style medicine lines, and common prescription formatting structures. It then isolates candidate medicine names and removes prompt artifacts, wrappers, or formatting noise. This layer ensures that medicine candidates are captured even when the model output is not perfectly structured.

After medicine identification, the architecture applies a medicine normalization and correction layer. This module standardizes medicine names through text normalization, alias lookup, conservative cleaning, and fuzzy matching. An alias dataset stores previously confirmed OCR-to-corrected-name mappings so that recurring extraction errors can be automatically corrected in future runs. A fuzzy matching mechanism compares uncertain medicine names with entries in the local medicine dataset and attempts to map distorted OCR forms to the nearest valid medicine candidate. In high-accuracy processing mode, the system can further invoke Gemini-based name correction to repair OCR errors when local matching is insufficient. This layer significantly improves robustness when dealing with handwriting or noisy prescription images.

The department-classification layer is the next core component. Its role is to assign each resolved medicine to one of the supported labels: Cardiology, Ophthalmology, or Unknown. The architecture implements a hierarchical decision strategy for this task. First, the system checks the curated local medicine dataset for exact or substring-based matches. If a compatible local machine-learning classifier is available, it may also be used for prediction. If the prediction confidence is weak or the local logic fails to identify the medicine confidently, the system can use Gemini fallback in high-accuracy mode. If none of the available methods can confidently determine the department, the architecture deliberately assigns the label Unknown instead of forcing a potentially incorrect department. This design reduces false classification while preserving uncertain cases for later review.

The architecture also contains a review and validation layer. This layer provides human-in-the-loop correction capability. After extraction and classification, the system generates review tables containing the original medicine text, corrected medicine name, department label, correction source, classification source, and confidence value. An additional unknown-only view highlights unresolved items. The user can manually modify corrected medicine names and department labels. When saved, these corrections update both the alias dataset and the medicine-department dataset. Therefore, this review layer not only improves the current result set but also enhances the future performance of the system by creating a feedback-driven learning loop.

Another important component is the dataset management layer. This layer handles local dataset files used by the system for medicine lookup, alias correction, and department mapping. The architecture allows users to upload additional CSV datasets containing medicine names and department labels. The system validates required columns, cleans the uploaded records, removes duplicates, merges them into the main dataset, and refreshes the local lookup cache. This dataset management layer allows the architecture to evolve continuously as users add local brand names, institution-specific medicine lists, or manually corrected entries.

The system also includes a result formatting and storage layer. After extraction and classification, the application produces final clean rows containing the fields medicine_name and department. These rows are used to generate downloadable outputs in multiple formats, including Excel or CSV spreadsheets, PDF reports, and ZIP packages for bulk processing. In bulk mode, the ZIP package can include combined outputs, summary files, and per-image CSV files. The architecture also supports a CSV collection subsystem that allows users to append results from multiple image-processing sessions into a selected CSV file. If the auto-append option is enabled, new processed results are automatically stored in the same CSV collection, simplifying longitudinal data accumulation.

The progress monitoring and state-management layer is also a key architectural element. The system uses Streamlit session state to persist important runtime information such as extracted text, uploaded file name, structured fields, medications list, processing status, export timestamps, bulk results, image hashes, and recent processing history. This enables the interface to remain responsive across reruns and allows the user to continue reviewing or exporting results after processing is complete. Progress bars and completion indicators are used to provide real-time feedback during single-image and bulk-image processing.

Finally, the overall architecture can be understood as a coordinated pipeline consisting of input acquisition, preprocessing, AI extraction, text parsing, medicine correction, department classification, result review, dataset learning, and export generation. The architecture is modular enough to support future extensions such as additional medical departments, stronger OCR modules, larger medicine datasets, model retraining, more advanced duplicate detection, and richer document-level analytics. In its current form, the architecture is particularly suited for prescription image analysis in environments where medicine detection, department classification, correction tracking, and structured dataset generation are all required within a single integrated application.
